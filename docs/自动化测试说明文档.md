# 自动化测试说明文档

## 文档目标
本文档用于说明当前项目中已补充的自动化测试内容，包括测试文件分布、覆盖范围、运行方式、环境依赖以及使用建议。

---

## 当前测试策略
本项目当前自动化测试采用两类组合方式：

### 1. 纯单元测试
特点：
- 不依赖数据库或外部服务
- 执行速度快
- 适合覆盖权限判断、状态机规则、参数校验等纯逻辑能力

### 2. 轻量集成测试
特点：
- 依赖本地 MySQL
- 使用真实数据模型和服务层调用
- 适合覆盖创建订单、分拣扫描、运输分配、路径优化等关键业务流程
- 测试数据会在用例结束后自动清理

### 3. HTTP Smoke / httptest 测试
特点：
- 基于 Gin 路由和 `httptest`
- 不依赖真实 HTTP 服务进程
- 用于快速验证核心路由是否可用、鉴权是否生效

## 各测试文件覆盖内容

### 1. 权限中间件测试
文件： [middleware/auth_test.go](../middleware/auth_test.go)

当前覆盖：
- 未携带 Token 时访问受保护接口应返回 401
- 合法 Token 能成功解析并把 `user_id`、`username`、`role` 写入上下文
- `RequirePermission` 能根据角色正确放行或拦截请求

### 2. 用户服务测试
文件： [services/user_service_test.go](../services/user_service_test.go)

当前覆盖：
- 注册参数校验
  - 用户名长度
  - 用户名字符合法性
  - 密码长度
  - 邮箱格式
  - 手机号格式
- 注册与登录集成流程
  - 注册成功
  - 登录成功
  - Token 返回正常
  - 返回用户信息中的密码字段被清空

### 3. 订单状态机测试
文件： [services/order_state_machine_test.go](../services/order_state_machine_test.go)

当前覆盖：
- 合法状态转换通过
- 非法角色执行状态转换被拦截
- 终态订单不能继续流转
- 异常状态的恢复流转存在
- 分拣中状态对司机可进入运输中与异常两种路径

### 4. 订单核心流程测试
文件： [services/order_service_test.go](../services/order_service_test.go)

当前覆盖：
- 订单参数校验
- 运费计算结果为正数
- 预计送达天数计算
- 创建订单
- 订单状态从待处理推进到已接单、已入库、分拣中
- 状态日志正常写入

### 5. 分拣流程测试
文件： [services/sorting_service_test.go](../services/sorting_service_test.go)

当前覆盖：
- 分拣规则匹配
- 分拣扫描成功
- 扫描后能命中规则并返回目标站点
- 分拣任务的 `sorted_count` 会自动更新

### 6. 运输分配与调度测试
文件： [services/transport_service_test.go](../services/transport_service_test.go)

当前覆盖：
- 批量车辆分配
- 分配后运输任务持久化成功
- 调度路径优化
- 优化结果中首尾站点保持不变
- 总距离结果为正数

### 7. API Smoke 测试
文件： [routes/smoke_test.go](../routes/smoke_test.go)

当前覆盖：
- `/health` 健康检查返回正常
- `/api/profile` 在未登录情况下返回 401

## 测试运行方式

### 1. 运行全部测试
```bash
go test ./...
```

### 2. 运行详细输出
```bash
go test -v ./...
```

### 3. 仅运行某一包测试
```bash
go test ./services
go test ./middleware
go test ./routes
```

### 4. 仅运行指定测试
```bash
go test -run TestRegisterAndLoginIntegration -v ./services
go test -run TestSortingRouteMatchAndScanFlowIntegration -v ./services
```

### 5. 使用 Makefile
```bash
make test
```

## 环境依赖说明

### 1. 纯单元测试
以下测试不依赖数据库：
- `middleware/auth_test.go`
- `services/order_state_machine_test.go`
- `routes/smoke_test.go`

### 2. 数据库集成测试
以下测试依赖本地 MySQL：
- `services/user_service_test.go`
- `services/order_service_test.go`
- `services/sorting_service_test.go`
- `services/transport_service_test.go`

当前测试工具会：
- 自动向上查找 `.env`
- 自动加载数据库配置
- 初始化 JWT 密钥
- 初始化数据库连接与自动迁移
- 在每个用例结束后清理相关测试数据

相关工具文件： [testutil/testenv.go](../testutil/testenv.go)

### 3. Redis说明
当前自动化测试未依赖 Redis，因此本地 Redis 未启动不会影响这批测试。

## 测试数据清理策略
测试工具会在 `t.Cleanup(...)` 中自动清理以下测试数据：
- 用户
- 订单
- 分拣规则 / 分拣记录 / 分拣任务
- 运输任务 / 车辆 / 配送记录
- 追踪记录 / 异常记录
- 站点流转记录 / 计划关联记录等

这可以避免测试运行后持续污染业务数据，但如果测试执行中断，仍可能留下少量临时数据，需要手动检查。

## 当前自动化测试价值

### 1. 防止核心逻辑回归
在继续开发追踪、异常、运输监控、前后端联调时，可以快速验证：
- 登录与权限是否被破坏
- 订单状态机是否被改坏
- 分拣扫描和任务进度是否正常
- 运输分配和调度算法是否仍然可用

### 2. 支撑后续重构
当前项目已进入后端收口阶段，后续 README 整理、接口统一、结构重构时，自动化测试可以提供最基本的安全网。

### 3. 降低重复手工验证成本
像权限、状态流转、路由匹配、任务创建这类反复回归的功能，不需要每次都重新用 Postman 手工点一遍。

### 4. 提高验收可信度
在项目汇报、验收或答辩中，除了展示功能可用，还可以说明：
- 关键后端模块已有自动化测试
- 测试可以重复执行
- 具备基本回归保障能力

---

## 当前局限
目前自动化测试仍然属于“第一版覆盖”，存在这些边界：
- 覆盖范围集中在核心流程，未覆盖所有接口分支
- 多数业务模块仍缺更细粒度异常场景测试
- HTTP 层目前只有基础 smoke test，还没有大范围 API 集成测试
- 仍未引入覆盖率统计与 CI 流程
