# 入库功能实现总结

## 功能概述

实现了完整的货物入库管理功能,包括入库扫描、入库记录创建、订单状态更新和入库记录查询。

## 实现内容

### 1. 数据模型

#### StationFlow 模型 (models/station.go)
```go
type StationFlow struct {
    ID            uint
    OrderID       uint      // 订单ID
    StationID     uint      // 站点ID
    FlowType      string    // 流转类型: in(入库) / out(出库)
    Quantity      int       // 数量
    Weight        float64   // 重量
    Volume        float64   // 体积
    OperatorID    uint      // 操作人ID
    NextStationID uint      // 下一站点ID
    Remark        string    // 备注
    CTime         int64     // 创建时间
    MTime         int64     // 修改时间
}
```

### 3. Service层 (services/warehouse_service.go)

#### InboundScan - 入库扫描
**功能**: 扫描订单进行入库操作

**业务逻辑**:
1. 验证订单是否存在
2. 验证站点是否存在且启用
3. 检查订单状态(只有已接单或运输中的订单可以入库)
4. 检查是否已在该站点入库(防止重复入库)
5. 创建入库流转记录
6. 更新订单状态为"已入库"(status=3)
7. 更新订单当前站点
8. 记录订单状态变更日志
9. 返回入库信息

**状态限制**:
- 允许入库的订单状态: 已接单(2)、运输中(5)
- 入库后订单状态: 已入库(3)

**防重复机制**:
- 检查同一订单在同一站点是否已有入库记录
- 如果已入库,返回错误提示

#### GetInboundRecords - 查询入库记录
**功能**: 分页查询入库记录

**查询条件**:
- 站点ID筛选
- 订单号筛选
- 时间范围筛选(开始时间、结束时间)
- 分页参数(page, page_size)

**返回信息**:
- 入库记录列表
- 订单信息(订单号、货物名称)
- 站点信息(站点名称)
- 操作人信息(操作人姓名)
- 货物信息(重量、体积)
- 入库时间

## 数据库字段统一问题修复

### 问题描述
在实现过程中发现`station_flows`表同时存在`c_time`、`m_time`和`ctime`、`mtime`四个字段。

### 问题原因
- 模型定义中的`CTime`和`MTime`字段没有明确指定`column`标签
- GORM的`autoCreateTime`标签会自动创建时间字段
- 同时GORM也会根据字段名创建蛇形命名的列
- 导致重复创建字段

### 解决方案
1. 删除数据库中重复的`ctime`和`mtime`字段:
```sql
ALTER TABLE station_flows DROP COLUMN ctime, DROP COLUMN mtime;
```

2. 在模型定义中明确指定列名:
```go
CTime int64 `gorm:"column:c_time;autoCreateTime;not null" json:"c_time"`
MTime int64 `gorm:"column:m_time;autoUpdateTime;not null" json:"m_time"`
```

3. 统一所有模型的时间字段定义,确保:
   - 数据库列名: `c_time`、`m_time`
   - JSON字段名: `c_time`、`m_time`
   - Go字段名: `CTime`、`MTime`
