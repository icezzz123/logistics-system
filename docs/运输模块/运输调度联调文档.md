# 运输调度联调文档

## 文档目标
本文档用于支持运输调度模块的接口联调、测试验证和演示准备，覆盖当前已实现的运输调度能力，包括路径优化、批次调度、调度建议、运输计划及订单加入计划等接口。

---

## 适用范围
当前调度相关接口位于 `/api/dispatch` 路由组下，均要求已登录并携带 `Authorization: Bearer <token>`。

已实现接口如下：

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/dispatch/route/optimize | 路径优化 |
| GET | /api/dispatch/batches | 批次调度列表 |
| POST | /api/dispatch/batches | 创建批次调度 |
| PUT | /api/dispatch/batches/:id/status | 更新批次调度状态 |
| POST | /api/dispatch/suggestion | 获取调度建议 |
| GET | /api/dispatch/plans | 运输计划列表 |
| POST | /api/dispatch/plans | 创建运输计划 |
| GET | /api/dispatch/plans/:id | 获取运输计划详情 |
| PUT | /api/dispatch/plans/:id | 更新运输计划 |
| PUT | /api/dispatch/plans/:id/status | 更新运输计划状态 |
| POST | /api/dispatch/plans/:id/orders | 将订单加入运输计划 |

---

## 联调前置条件

### 1. 鉴权要求
- 先调用登录接口 `/api/auth/login` 获取 JWT Token。
- 所有 `/api/dispatch/*` 接口都要求在请求头中带上：
  - `Authorization: Bearer <token>`

### 2. 基础数据准备
联调前建议先准备以下数据：
- 至少 2 个有效站点，用于路径优化。
- 至少 1 辆可用车辆，要求 `vehicles.status = 1`。
- 至少 1 名有效司机，要求用户角色 `role >= 3`。
- 至少 1 批待调度订单。
- 如果要验证载重逻辑，建议准备不同重量的订单。

### 3. 关键业务约束
- 创建批次调度时会校验车辆可用性、司机有效性、订单是否存在、总重量是否超出车辆载重。
- 创建批次调度后，系统会尝试为其中尚无有效运输任务的订单自动创建运输任务。
- 获取调度建议时，系统会基于可用车辆容量做贪心装箱分配。
- 创建运输计划时会校验车辆可用、司机有效。
- 将订单加入运输计划时会校验计划状态和载重上限。

## 核心状态说明

### 批次调度状态
| 状态值 | 说明 |
|------|------|
| pending | 待发车 |
| dispatched | 已发车 |
| in_transit | 运输中 |
| completed | 已完成 |
| cancelled | 已取消 |

### 运输计划状态
| 状态值 | 说明 |
|------|------|
| draft | 草稿 |
| confirmed | 已确认 |
| executing | 执行中 |
| completed | 已完成 |
| cancelled | 已取消 |

## 推荐联调顺序

### 流程一：路径优化联调
适用于验证站点路径排序能力。

1. 准备多个站点 ID。
2. 调用 `/api/dispatch/route/optimize`。
3. 核对返回的原始顺序、优化顺序、总距离、节省距离、预计耗时。

### 流程二：批次调度联调
适用于验证车辆、司机、订单的批量调度能力。

1. 准备车辆、司机、订单。
2. 调用 `/api/dispatch/batches` 创建批次调度。
3. 核对返回的 `batch_no`、`order_count`、`total_weight`、关联运输任务列表。
4. 调用 `/api/dispatch/batches` 列表接口验证分页与筛选。
5. 调用 `/api/dispatch/batches/:id/status` 更新批次状态。

### 流程三：调度建议联调
适用于验证系统自动分配建议。

1. 准备多个待调度订单。
2. 确保存在多辆可用车辆。
3. 调用 `/api/dispatch/suggestion`。
4. 核对建议批次、每车订单列表、装载率、未分配订单。

### 流程四：运输计划联调
适用于验证计划管理与载重控制。

1. 调用 `/api/dispatch/plans` 创建运输计划。
2. 调用 `/api/dispatch/plans` 查询计划列表。
3. 调用 `/api/dispatch/plans/:id` 查看详情。
4. 调用 `/api/dispatch/plans/:id/orders` 加入订单。
5. 调用 `/api/dispatch/plans/:id/status` 更新计划状态。
6. 必要时调用 `/api/dispatch/plans/:id` 更新计划信息。

## 接口联调示例

### 1. 登录获取 Token
```powershell
$loginBody = @{
    username = 'admin'
    password = 'Pass1234'
} | ConvertTo-Json

$loginResp = Invoke-RestMethod -Method Post `
    -Uri 'http://127.0.0.1:8080/api/auth/login' `
    -ContentType 'application/json' `
    -Body $loginBody

$token = $loginResp.data.token
$headers = @{ Authorization = "Bearer $token" }
```

### 2. 路径优化
```powershell
$body = @{
    station_ids = @(1, 2, 3, 4)
    vehicle_id  = 1
} | ConvertTo-Json

Invoke-RestMethod -Method Post `
    -Uri 'http://127.0.0.1:8080/api/dispatch/route/optimize' `
    -Headers $headers `
    -ContentType 'application/json' `
    -Body $body
```

### 3. 创建批次调度
```powershell
$body = @{
    batch_name   = '华东干线测试批次'
    vehicle_id   = 1
    driver_id    = 4
    order_ids    = @(31, 32, 33)
    planned_time = [int64]([DateTimeOffset]::Now.ToUnixTimeSeconds() + 3600)
    remark       = '联调测试批次'
} | ConvertTo-Json

Invoke-RestMethod -Method Post `
    -Uri 'http://127.0.0.1:8080/api/dispatch/batches' `
    -Headers $headers `
    -ContentType 'application/json' `
    -Body $body
```

### 4. 查询批次调度列表
```powershell
Invoke-RestMethod -Method Get `
    -Uri 'http://127.0.0.1:8080/api/dispatch/batches?page=1&page_size=10&status=pending' `
    -Headers $headers
```

### 5. 更新批次状态
```powershell
$body = @{
    status = 'dispatched'
    remark = '联调更新为已发车'
} | ConvertTo-Json

    -Uri 'http://127.0.0.1:8080/api/dispatch/batches/1/status' `
    -Headers $headers `
    -ContentType 'application/json' `
    -Body $body
```

### 6. 获取调度建议

    date      = [int64]([DateTimeOffset]::Now.ToUnixTimeSeconds())

## 联调检查点

### 路径优化检查点
- 返回的 `optimized_order` 数量应与输入站点数一致。
- `saved_distance` 不应为负数。
- `estimated_time` 应大于等于 0。

### 批次调度检查点
- `batch_no` 应自动生成。
- `order_count` 与请求订单数一致。
- `total_weight` 应等于订单总重量。
- 若车辆容量不足，应返回明确错误。
- 若订单已存在有效运输任务，系统会跳过该订单的任务创建。

### 调度建议检查点
- `suggestions` 中的车辆应为可用状态。
- `load_rate` 应与订单总重和车辆容量匹配。
- 容量不足的订单应进入 `unassigned_orders`。

### 运输计划检查点
- `plan_no` 应自动生成。
- 加入订单后 `used_capacity` 应更新。
- 计划状态变更后列表与详情查询应同步变化。
- 当计划已完成或已取消时，不应再允许加入订单。

## 常见联调问题

### 1. 返回“车辆不存在或不可用”
排查项：
- 对应 `vehicle_id` 是否存在。
- 车辆状态是否为 `1`。

### 2. 返回“司机不存在或权限不足”
排查项：
- `driver_id` 对应用户是否存在。
- 用户角色是否满足 `role >= 3`。

### 3. 返回“部分订单不存在”
排查项：
- 请求中的 `order_ids` 是否包含已删除或错误 ID。

### 4. 返回“订单总重量超过车辆载重”
排查项：
- 批次总重量是否超出 `vehicles.capacity`。
- 车辆载重是否正确录入。

### 5. 返回“添加后总重量超过计划最大载重”
排查项：
- `transport_plans.max_capacity` 是否设置过小。
- 已有 `used_capacity` 是否已接近上限。

---

## 推荐联调策略
- 先联调路径优化和调度建议，这两类接口对数据库写入影响较小，便于快速验证。
- 再联调批次调度和运输计划，这两类接口会创建或更新实际业务数据。
- 联调批次调度时，建议先使用全新订单，避免受已有运输任务数据影响。
- 若用于答辩演示，建议提前准备一组固定站点、车辆、司机和订单数据，保证每次演示输出稳定。
